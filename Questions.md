# Питання та відповіді — Паралельне програмування

Контрольні питання до лабораторних робіт з курсу.

---

## Лабораторна 1. Робота з задачами та потоками

### 1. Що таке задача?

Задача (`Task`) — це одиниця роботи, яку розробник описує і передає runtime'у на виконання. У потоці виконується певна частина задачі, причому виконання може початися в одному потоці, а завершитися в іншому — планувальник сам керує потоками. Розробник оперує задачами, а ОС оперує потоками.

### 2. Чим відрізняється виконання багатопоточної програми від однопоточної?

Однопоточна виконує інструкції строго послідовно в одному потоці виконання. Багатопоточна — паралельно (на різних ядрах) або з перемиканням між потоками планувальником ОС, що дозволяє: не блокувати UI під час тривалих обчислень, використовувати кілька ядер, виконувати незалежні задачі одночасно. Натомість потрібна синхронізація доступу до спільних ресурсів і безпечне оновлення UI лише з основного потоку (`MainThread.BeginInvokeOnMainThread`).

### 3. Як створити і запустити задачу?

Кілька варіантів з прикладів 1.1 і 1.3 методички:

- `var t = new Task(method); t.Start();` — створити, потім запустити (приклад 1.1).
- `Task.Factory.StartNew(method)` — створити і запустити одразу.
- `Task.Run(method)` — спрощений, найпоширеніший спосіб.
- З параметром стану: `new Task(action, state)` або `Task.Factory.StartNew(action, state)`, де метод приймає `object?`.

### 4. Як завершити потік?

Кооперативно через `CancellationTokenSource`: зовні викликається `cts.Cancel()`; усередині задача періодично перевіряє `token.IsCancellationRequested` або кидається `OperationCanceledException` від таких методів як `Task.Delay(ms, token)`. Задача коректно повертає управління — це і є її завершення. Примусове знищення (`Thread.Abort`) у сучасному .NET не підтримується, бо лишає ресурси у непередбачуваному стані.

### 5. Як передати аргумент потоку?

Два способи з прикладу 1.3:

- **Через лямбду** (захоплення змінних): `new Task(() => PrintIntegers(id, max))`.
- **Через параметр state** (`object?`): `new Task(PrintIntegersWithState, Tuple.Create("id", max))` — у методі робиться `state is Tuple<string,int> t` і далі використовуються `t.Item1`/`t.Item2`.

### 6. Пріоритети потоків. Як поміняти пріоритет потоку?

Для потоку — властивість `Thread.Priority` зі значеннями `Lowest`, `BelowNormal`, `Normal`, `AboveNormal`, `Highest` (підказка планувальнику ОС, не гарантія). Для процесу — `process.PriorityClass = ProcessPriorityClass.BelowNormal` (значення: `Idle`, `BelowNormal`, `Normal`, `AboveNormal`, `High`, `RealTime`). Реальний розподіл часу залежить від планувальника ОС.

### 7. Який метод в новому потоці запускає метод Start() класу Task?

Той метод (делегат `Action`/`Action<object?>`), який було передано в конструктор `Task`. У прикладі 1.1: `new Task(taskExample.Run)` — `Start()` запускає `taskExample.Run` у потоці пулу.

### 8. Що робить метод Wait() класу Task?

Блокує **поточний** (викликаючий) потік до завершення задачі. У прикладі 1.1 основний потік викликає `task.Wait()` щоб дочекатися фонової задачі перед виведенням фінального значення. Перевантаження приймають таймаут і `CancellationToken`. Якщо задача завершилась з винятком, `Wait()` кине `AggregateException`.

## Лабораторна 2. Процеси

### 1. Що таке процес?

Процес — це екземпляр програми, що виконується. Він має власний адресний простір пам'яті, набір ресурсів (дескриптори файлів, мережеві сокети, обробники), контекст безпеки і власний набір потоків. ОС планує процеси на ядрах CPU, а кожен процес містить хоча б один потік. Процеси ізольовані один від одного — пам'ять одного процесу недоступна іншому без явного механізму (named pipes, м'ютекси, shared memory тощо).

### 2. Який клас і який простір імен використовується для створення процесу?

Клас — **`Process`**, простір імен — **`System.Diagnostics`**.

### 3. Як створити і запустити процес?

Три способи з методички:

- **Найпростіший** (лістинг 2.1.1): `Process.Start("notepad.exe")` — статичний метод запускає виконуваний файл.
- **З аргументами** (2.1.2): `Process.Start("notepad.exe", arguments: filePath)`.
- **Через конфігурацію** (2.1.3) — найгнучкіший: створити `ProcessStartInfo` (вказати `FileName`, `Arguments`, `UseShellExecute`, перенаправлення потоків, змінні середовища і т.д.), потім `var p = new Process { StartInfo = startInfo }; p.Start();`.

### 4. Як зупинити виконання процесу?

- **`process.Kill()`** — примусово завершує процес (лістинг 2.1.9). Приклад: знайти процеси за іменем (`Process.GetProcessesByName("notepad")`) і викликати `Kill()` на кожному.
- **`process.CloseMainWindow()`** — м'яке закриття (надсилає WM_CLOSE головному вікну, користувач може не підтвердити збереження). Стандартний підхід для GUI-процесів.

### 5. Як визначити завершення процесу?

Два способи з лістингу 2.1.4:

- **Блокуюче**: `process.WaitForExit()` — поточний потік чекає поки процес не завершиться. Підходить для консольних утиліт; у GUI **заморозить інтерфейс**.
- **Подієве (асинхронне)**: `process.EnableRaisingEvents = true; process.Exited += handler;` — обробник викликається з пула потоків коли процес завершиться. Підходить для GUI; з UI-потоку перейти через `MainThread.BeginInvokeOnMainThread`.

### 6. Що запускає метод Start() класу Process?

Запускає **виконуваний файл** (окрему програму, новий ОС-процес), описаний у `process.StartInfo.FileName` з аргументами з `Arguments`. На відміну від `Task.Start()`, який запускає **метод-делегат** в потоці того самого процесу, `Process.Start()` створює **повністю окремий процес ОС** з власною пам'яттю, ID і ресурсами. Повертає `bool`: `true` якщо створено новий процес, `false` якщо запит обробив існуючий процес (наприклад, друга інстанція Excel приєдналась до першої).

## Лабораторна 3. Синхронізація

### 1. Що таке синхронізація?

Координація виконання потоків (або процесів), що працюють зі спільними ресурсами, щоб уникнути непередбачуваних результатів через одночасний доступ. Синхронізація встановлює правила: коли потік може ввійти в критичну секцію, коли почекати, коли продовжити.

### 2. Навіщо і коли потрібна синхронізація? Приклади.

Потрібна, коли:

- **Гонитва даних (race condition)** — кілька потоків пишуть/читають спільну змінну. Приклад 3.1 з методички: 100 задач операціюють з балансом рахунку; без `lock` фінальний баланс непередбачуваний (втрачені оновлення).
- **Сигналізація** — потік чекає коли інший зробить роботу. Приклад 3.3: головний потік чекає, поки дочірній досягне середини обчислень.
- **Обмеження доступу** — наприклад, лише один потік пише у файл за раз; або лише N потоків одночасно виконують ресурсоємну операцію (приклад 3.5 — семафор).
- **Узгоджений знімок** — щоб читач побачив усі зміни цілісно, а не половину старих, половину нових (приклад 3.2 з мютексом між процесами).

### 3. Які існують об'єкти синхронізації?

- `lock` — синтаксис над `Monitor` (приклад 3.1).
- `Monitor` — Enter/Exit/Wait/Pulse.
- `Mutex` — взаємне виключення; може бути іменованим і **працює між процесами** (приклад 3.2).
- `Semaphore` / `SemaphoreSlim` — лічильник пропусків (приклад 3.5).
- `AutoResetEvent` / `ManualResetEvent` / `EventWaitHandle` — сигнальні об'єкти (приклади 3.3, 3.4).
- `ReaderWriterLockSlim` — окремі замки для читачів і писачів.
- `Barrier`, `CountdownEvent` — координація етапів кількох потоків.
- `Interlocked` — атомарні операції без замків.
- `SpinLock` — швидкий замок для дуже коротких секцій.

### 4. Що таке м'ютекс?

**Mut**ual **ex**clusion — об'єкт, яким одночасно володіє лише ОДИН потік (або процес). Інші чекають у `WaitOne()` поки власник не викличе `ReleaseMutex()`. Має поняття "власника" — звільнити може тільки той потік, що захопив. Іменований мютекс працює крос-процесно (приклад 3.2: батьківський і дочірній процеси чергуються у виводі).

### 5. Що таке семафор?

Лічильник доступу: дозволяє максимум N одночасних "власників". `WaitOne()` зменшує лічильник на 1 (блокує, якщо 0); `Release()` збільшує. На відміну від мютексу **не має поняття власника** — звільняти може будь-який потік. У прикладі 3.5 семафор з ємністю 1 використовується як одно-місна черга: дочірній потік пише значення → `Release()`; головний `WaitOne()` → читає.

### 6. Що таке події?

`EventWaitHandle` (зокрема `AutoResetEvent` та `ManualResetEvent`) — сигнальні об'єкти з двома станами: signaled / non-signaled. У сигнальному стані `WaitOne()` пропускає; у несигнальному — блокує. Перемикання: `Set()` → signaled, `Reset()` → non-signaled.

- `AutoResetEvent` — пропустивши одного очікувача, **автоматично** скидається (приклад 3.3).
- `ManualResetEvent` — лишається у Set поки `Reset()` не викликати явно; пропускає **усіх** очікувачів (приклад 3.4).

### 7. Методи класу Monitor що використовуються для синхронізації

- `Monitor.Enter(obj)` / `Monitor.Exit(obj)` — захоплення/звільнення замка (`lock(obj){}` — синтаксичний цукор над цим).
- `Monitor.TryEnter(obj, timeout)` — захоплення з таймаутом.
- `Monitor.Wait(obj)` — атомарно звільняє замок та блокується, очікуючи `Pulse`/`PulseAll`.
- `Monitor.Pulse(obj)` — будить **один** очікувач у `Wait`.
- `Monitor.PulseAll(obj)` — будить **усіх** очікувачів.
- `Monitor.IsEntered(obj)` — чи поточний потік уже володіє замком.

### 8. Чим семафор відрізняється від події?

Принципово різні концепції:

- **Семафор** — лічильник: "залишилось N пропусків". `WaitOne()` "споживає" один (декремент); коли 0 — блокується. Кожен `Release()` додає пропуск назад.
- **Подія** — сигнал стану: "ВСТАНОВЛЕНО / СКИНУТО". Не має лічильника. У сигнальному стані `Set` усі (Manual) або один (Auto) очікувач проходять.

Аналогія: семафор — пакет з 5 квитків у кіно (закінчилися — чекай), подія — стартовий пістолет на гонці (всі чекають пострілу, потім біжать).

## Лабораторна 4. Відображення файлу у пам'яті

### 1. Концепція механізму відображення файлів у пам'яті

Memory-mapped file — це механізм, при якому регіон файлу (або просто блок віртуальної пам'яті) проектується в адресний простір процесу. Звернення до нього виглядає як до звичайного масиву в пам'яті, а ОС прозоро підкачує сторінки з диска через page fault. Кілька процесів можуть мапити один і той же файл/регіон → отримують **спільну пам'ять** (фактично IPC без копіювання). Переваги: відсутність явних `Read`/`Write` викликів, ефективна робота з великими файлами, можливість випадкового доступу через адресацію.

### 2. Назвати два типи зіставлених у пам'яті файлів і їх відмінність

- **Постійний (persistent)** — прив'язаний до реального файлу на диску. Створюється через `CreateFromFile`. Зміни синхронізуються на диск, дані переживають перезавантаження. Приклад 4.1 з методички.
- **Непостійний (non-persistent)** — існує лише у RAM, без бекінга на диску. Створюється через `CreateNew`. Зникає коли всі процеси закриють свої посилання. Призначення — швидкий IPC між процесами через спільну пам'ять. Приклади 4.2 і 4.3.

### 3. Який метод використовується для отримання об'єкта, який представляє постійний зіставлений у пам'яті файл?

`MemoryMappedFile.CreateFromFile(path, mode, mapName, capacity)`. Приклад 4.1 з методички:

```csharp
MemoryMappedFile.CreateFromFile("a1.dta", FileMode.OpenOrCreate, "file", 1);
```

### 4. Який метод використовується для отримання об'єкта, який представляє непостійний зіставлений у пам'яті файл?

`MemoryMappedFile.CreateNew(mapName, capacity)` — створення в процесі-власнику. Приклад 4.2:

```csharp
MemoryMappedFile.CreateNew("n_file", 4096);
```

Інший процес для доступу до того самого MMF використовує `MemoryMappedFile.OpenExisting(mapName)` (приклад 4.3 — процес B).

### 5. Який метод використовується для отримання об'єкта представлення для послідовного і довільного доступу до зіставленого в пам'яті файлу?

- **Послідовний доступ** — `CreateViewStream()`, що повертає `MemoryMappedViewStream`. Над цим потоком використовуються `BinaryReader`/`BinaryWriter`/`StreamReader`. Це те, що застосовується у всіх прикладах методички (4.1–4.3): `using var stream = mmf.CreateViewStream();`.
- **Довільний (random) доступ** — `CreateViewAccessor()`, що повертає `MemoryMappedViewAccessor`. Дозволяє читати/писати типізовані значення у будь-яку позицію через `Read<T>(position)` / `Write<T>(position, value)`.
